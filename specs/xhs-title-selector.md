# 小红书标题策略 Skill 设计文档

## 1. 目标概述

这个 Skill 用于把“小红书标题”从临场拍脑袋，变成一个可追溯、可匹配、可解释的标题策略系统。

它要解决的不是“让 AI 随便起几个标题”，而是帮助用户稳定回答下面这些问题：

- 这个内容更适合哪种标题触发器
- 标题应该打反常识、好奇心、损失规避，还是身份代入
- 当前标题的问题是公式选错了，还是执行得不够好
- 标题到底应该吸引谁，而不是只求更炸
- 哪几个标题是当前内容里真正值得优先测试的

## 2. 设计来源

这个 Skill 明显受到 `dbskill` 里的 `dbs-xhs-title` 启发，但会做更适合 Cell 仓库的改造。

保留的部分：

- 标题的核心不是生成，而是匹配
- 每个标题都应该能追溯到具体公式
- 先判断内容属性，再选触发器
- 输出不只给标题，还要说明为什么选这个公式
- 支持“已有标题优化”和“只浏览公式”两种分支

改造的部分：

- 不照搬原始 75 条目录，而是做一套 Cell 仓库可维护的精炼公式库
- 不把标题和平台玄学混在一起，而是强调内容 promise、目标读者和触发器匹配
- 更明确地和 `content-strategy-diagnosis`、`hook-optimizer` 分工

## 3. 技能定位

它是一个“标题策略 / 标题公式匹配 Skill”，不是一个通用文案生成器。

## 4. 与现有 Skill 的关系

- `content-strategy-diagnosis`
  - 负责判断选题、载体、平台和内容目标是否成立
- `xhs-title-selector`
  - 负责在题已经相对成立的前提下，选择最合适的小红书标题公式
- `hook-optimizer`
  - 负责标题选定后，再优化内容开头
- `deep-writer`
  - 负责正文结构和内容

理想顺序通常是：

1. 先判断内容值不值得做
2. 再用 `xhs-title-selector` 做标题策略
3. 再用 `hook-optimizer` 和其他 skill 完成正文承接

## 5. 建议技能名

建议英文目录名：

`xhs-title-selector`

建议展示名：

`小红书标题策略`

## 6. 适用场景

这个 Skill 主要适用于：

- 已有一个内容话题，想起更合适的小红书标题
- 已经有正文或选题，想看适合打哪种心理触发器
- 已有一个标题，但怀疑公式选错了
- 想先浏览标题家族，再决定走哪一路
- 想要一组可测试的标题方案，而不是一个单点答案

## 7. 非目标

V1 默认不解决以下问题：

- 在没有任何话题或内容方向时，无边界批量起标题
- 把标题当成独立于正文之外的点击作弊工具
- 代替完整选题诊断
- 代替平台运营策略

## 8. 核心用户问题

这个 Skill 需要稳定回答以下问题：

1. 这个内容到底该打哪种标题机制？
2. 为什么这个标题比另一个更适合当前内容？
3. 当前标题是公式选错了，还是只差一层打磨？
4. 标题该更偏点击、共鸣、筛选人群，还是承诺结果？
5. 哪几个标题适合先拿去 A/B 测试？

## 9. 典型触发语句

- 帮我起个小红书标题
- 这个内容适合什么标题公式
- 小红书标题怎么写
- 我这个标题不太行，帮我看公式是不是错了
- 先别写正文，先把标题跑出来

## 10. 默认输入约定

### 10.1 必填输入

- 至少提供以下之一：
  - 选题
  - 行业 / 领域
  - 一段正文或内容说明
  - 一个已有标题

### 10.2 选填输入

- 目标人群
- 内容目标：点击 / 涨粉 / 建认知 / 咨询 / 成交
- 语气偏好：克制 / 反常识 / 强冲突 / 更实用
- 是否需要多个方向并行测试

### 10.3 默认假设

如果用户没说明，默认：

- 输出语言为中文
- 不是自由生成，而是公式匹配
- 至少覆盖 3 个不同触发器家族
- 每个标题都要标注公式编号
- 最后给出 Top 3 推荐和测试优先级

## 11. 输出契约

### 11.1 标准版

至少包含：

- 话题与目标人群判断
- 匹配到的标题家族
- 5-8 个标题方案
- 每个方案的公式编号与理由
- Top 3 推荐

### 11.2 已有标题优化版

至少包含：

- 当前标题所用公式判断
- 问题诊断
- 同公式优化版
- 替代公式方案

## 12. 方法论原则

### 12.1 标题不是生成题，是匹配题

真正的关键不是“AI 会不会起标题”，而是“你到底该用哪种标题机制”。

### 12.2 标题必须服务于内容 promise

如果正文 promise 不清，标题也会虚。

### 12.3 公式要可追溯

每一个结果都要能回到某个家族和编号，不做不可解释的自由发挥。

### 12.4 多样性比同义改写更重要

一组标题应该覆盖不同触发器，而不是 8 个几乎一样的版本。

### 12.5 标题必须可点击，也必须可兑现

如果标题把答案说完了，或正文根本接不住，再好看的标题也没用。

## 14. V1 关键约束

1. 不做无来源的自由标题生成
2. 不能只给一个方向，必须给可比较方案
3. 不能脱离正文 promise 去做纯 clickbait
4. 必须区分“公式错了”和“执行差了”
5. 必须和现有内容诊断 / 开头优化 skill 明确分工
