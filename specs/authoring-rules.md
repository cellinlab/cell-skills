# Skill 编写约定

这份文档约束这个仓库里 `specs/` 和 `skills/` 的写法边界，目标是让设计信息和运行信息分层，减少 Skill 在实际运行时产生幻觉的机会。

## 1. 总原则

把“为什么这样设计”留在 `specs/`，把“运行时该怎么做”留在 `skills/`。

不要把设计复盘、来源说明、上游对比写进运行层。

## 2. `specs/` 里可以写什么

`specs/` 是设计文档层，可以出现这些内容：

- 设计来源
- 上游仓库或参考 Skill
- 学习对象
- 保留的部分 / 改造的部分
- 为什么拆分或合并
- 和仓库内其他 Skill 的设计分工
- 未来版本规划

这些内容的读者是维护者，不是运行中的 Agent。

## 3. `skills/` 里应该写什么

`skills/` 下的 `SKILL.md`、`references/`、`assets/` 说明，应该只保留运行时真正需要的信息：

- 适用场景
- 非目标
- 输入假设
- 工作流
- 输出契约
- 硬约束
- 故障处理
- 资源导航

如果某条信息不能帮助 Agent 更稳地完成当前任务，就不要放进实现层。

## 4. `skills/` 里不要写什么

以下内容默认不应该出现在 `skills/` 的实现层：

- “这个 Skill 吸收了……”
- “这个 Skill 参考了……”
- “这个 Skill 学习自……”
- “上游 repo / 上游 Skill / 来源仓库……”
- “保留的部分 / 改造的部分……”
- 任何仅用于解释历史来源、设计出处、演化路径的叙述

这些内容会让 Agent 在运行时更容易把历史上下文误当成当前事实。

## 5. 允许的例外

有些外部信息属于运行依赖，而不是来源叙述，这类内容可以保留在 `skills/`：

- 当前任务真正需要调用的外部 API
- 当前任务必须访问的官方平台或文档入口
- 用户必须完成的真实配置步骤
- 当前仓库内其他 Skill 的协作指向

判断标准是：

- 如果删掉这段内容，Agent 会更难完成任务，那它可能应该留在 `skills/`
- 如果删掉这段内容，只会失去“这个 Skill 从哪学来的”背景，那它应该放到 `specs/`

## 6. 写作建议

当你发现自己想写下面这类句子时，优先改写成运行导向表述：

- 不要写：“这个 Skill 从某某仓库抽出了共同部分”
- 改写成：“目标是提供一条稳定、可迁移的最小闭环”

- 不要写：“两个上游实现都这样做”
- 改写成：“更稳的默认顺序通常是……”

- 不要写：“这里和上游实现不同”
- 改写成：“不同项目在这里可能有差异，优先复用现有稳定实现”

## 7. 评审清单

新增或修改 Skill 时，至少检查一次：

```bash
python3 scripts/check-skill-doc-boundaries.py
```

这个脚本会只检查 `skills/` 运行层文档，自动跳过 markdown 代码块，并把命中的来源叙述按文件和行号报出来。

如果脚本报错，优先判断命中的内容是不是“来源叙述”。

是的话，移到对应的 `specs/` 文档里。

需要做第一轮粗扫时，再用：

```bash
rg -n "设计来源|上游|吸收|参考了|学习自|保留的部分|改造的部分|源于|借鉴" skills
```

## 8. pre-commit

这个仓库把边界检查接进了 `.pre-commit-config.yaml`。

首次启用：

```bash
pip install pre-commit
pre-commit install
```

手动全量运行：

```bash
pre-commit run --all-files
```

当前接入的关键检查：

- `skill-doc-boundaries`
  - 调用 `python3 scripts/check-skill-doc-boundaries.py`
  - 阻止把“设计来源 / 上游参考 / 保留的部分 / 改造的部分”这类叙述写进 `skills/` 运行层
