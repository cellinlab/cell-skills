# 超级个体 / 一人公司案例调研 Skill 设计文档

## 1. 目标概述

本 Skill 用于对“超级个体 / 超级 IP / 一人公司”相关的人物或案例进行系统化公开信息调研，并将结果整理为中文输出，供研究、学习、拆解与方法提炼使用。

它要解决的不是“帮用户写一篇人物简介”，而是帮助用户完成一套可追溯、可比较、可复用的案例研究流程，尤其聚焦以下三个核心维度：

- 内容创作策略
- IP 打造策略
- 商业模式与变现闭环

## 2. 适用场景

这个 Skill 主要面向以下使用场景：

- 调研某个具体人物是否值得纳入“超级个体案例库”
- 拆解某位创作者 / 主理人 / 顾问 / 一人公司经营者的内容与商业路径
- 为长文、研究报告、播客选题、栏目内容提供结构化资料
- 提炼可复制动作、关键节点和方法论
- 为后续横向对比不同案例做标准化铺垫

## 3. 非目标

V1 不解决以下问题：

- 私人信息挖掘或非公开情报搜集
- 纯八卦向人物介绍
- 无证据支撑的收入、团队、合作猜测
- 多案例横向数据库系统的自动化构建
- 投资建议、法律建议或尽职调查级别结论

## 4. 建议技能名

建议将技能英文文件夹名定为：

`opc-case-research`

理由：

- 名称更短，更适合长期使用与引用
- `OPC` 已有一定认知基础，适合作为个人品牌语境中的简称
- 具体含义可以通过 Skill description 补充为 `one-person company`
- 命名长度适中，符合 Skill 命名规范

后续对外展示名可使用：

`超级个体案例调研`

## 5. 核心用户问题

这个 Skill 需要稳定回答下面几类问题：

1. 这个人 / 这个案例到底是谁，为什么值得研究？
2. TA 是如何一步步形成当前定位、内容体系和商业闭环的？
3. TA 的内容策略、渠道策略和品牌策略分别是什么？
4. TA 的收入结构大致由什么组成，哪些是稳定现金流，哪些是品牌势能？
5. 对想成为超级个体的人来说，哪些动作可复制，哪些高度依赖时机、资历和个人禀赋？

## 6. 典型触发语句

未来这个 Skill 应该能自然响应类似需求：

- 帮我调研 XXX，重点看他的内容创作、IP 打造和商业模式
- 用中文系统拆解一下 XXX 为什么能成为一人公司案例
- 我想研究 XXX 是否适合作为超级个体样本，请先做案例速览
- 帮我做一份 XXX 的公开资料调研，重点提炼可复制的方法
- 调研 XXX，给我一份结构化报告，不要写成人物简介

## 7. 默认输入约定

### 7.1 必填输入

- 调研对象：人物名、品牌名或案例名

### 7.2 选填输入

- 研究重点：内容 / IP / 商业模式 / 渠道 / 方法论 / 关键节点
- 输出深度：快速版 / 标准版 / 深度版
- 时间范围：近一年 / 全部公开资料 / 某个阶段
- 语言范围：中文优先 / 中英双语资料都看
- 对比对象：是否需要与其他案例比较

### 7.3 默认假设

如果用户没有额外说明，Skill 默认采用以下假设：

- 使用公开信息
- 结果用中文整理
- 重点放在内容创作、IP 打造、商业模式
- 先做单案例研究
- 结论必须区分事实、推断、未知

## 8. 输出契约

这个 Skill 的输出不应只是一篇大段文字，而应是一组结构化研究结果。

### 8.1 快速版

适用于初筛，至少包含：

- 案例一句话定位
- 核心身份标签
- 主平台与主要内容形式
- 可见商业入口
- 值得研究的 3-5 个理由
- 待验证问题

### 8.2 标准版

适用于大多数研究场景，至少包含：

- 案例速览卡
- 来源地图
- 关键时间线
- 内容与 IP 拆解
- 渠道矩阵
- 商业模式结构拆解
- 可复制性分析
- 待验证信息与局限

### 8.3 深度版

适用于正式研究稿，除标准版外还应尽量补齐：

- 结构化证据库
- 内容抽样记录
- 关键决策节点复盘
- 收入结构区间估算与不确定性披露
- 可视化附录建议

## 9. 结果风格要求

输出风格应满足以下要求：

- 中文写作
- 结构化强，便于沉淀和复用
- 优先研究与经营视角，而非粉丝视角
- 强调证据、时间点和不确定性
- 能提炼方法，而不是只做赞美

## 10. 方法论原则

这个 Skill 的底层原则直接继承你现有的人肉调研框架，但在实现上需要收敛为可执行规则。

### 10.1 信息边界

- 只使用公开信息
- 优先本人表达、官方页面、原始资料
- 对敏感、隐私、未证实信息一律不采纳

### 10.2 证据优先级

- A 级：本人公开表达、官网、原始节目页、官方活动页
- B 级：主流媒体、平台人物页、行业数据库、合作方官方介绍
- C 级：转载、整理帖、论坛、评论区，仅作线索

### 10.3 信息状态

所有关键结论应尽量标注为：

- 事实
- 推断
- 未知

### 10.4 核验要求

- 关键事实尽量双源交叉
- 原始来源优先于二手整理
- 遇到冲突信息要显式处理
- 涉及收入、团队、规模时必须保守

## 11. Skill 工作流设计

这个 Skill 的执行流程建议固定为 7 个阶段。

### 阶段 1：明确对象与调研范围

目标：

- 确认调研对象是谁
- 明确研究重点与输出深度
- 如果用户信息不足，先建立基础档案卡

输出：

- 调研对象确认
- 初始问题框架
- 默认假设说明

### 阶段 2：建立基础档案与来源地图

目标：

- 快速建立身份、平台、官网、赛道、商业入口等基础信息
- 整理 A/B/C 三层来源地图

输出：

- 基础档案卡
- 来源清单
- 优先阅读顺序

### 阶段 3：建立时间线骨架

目标：

- 先建立事件顺序感，再进入分析
- 避免没有时间感就直接下观点

输出：

- 时间线 V1
- 关键节点列表

### 阶段 4：拆解内容与 IP

目标：

- 识别核心主张、长期主题、内容形式、风格与角色感
- 判断主阵地、分发阵地与导流关系

输出：

- IP 定位结论
- 内容结构概览
- 渠道矩阵

### 阶段 5：拆解商业模式

目标：

- 识别收入来源、产品化路径、服务化路径、品牌合作路径
- 判断经常性收入、一次性收入与品牌势能型收益

输出：

- 商业模式结构图式说明
- 主收入与辅助收入判断
- 风险集中度分析

### 阶段 6：复盘关键决策节点

目标：

- 解释转型、迁移、升级背后的决策意义
- 把“发生了什么”升级为“为什么重要”

输出：

- 关键决策复盘表
- 决策前后变化说明

### 阶段 7：提炼可复制性

目标：

- 输出对超级个体研究真正有价值的方法启发
- 区分普通人可学、需要积累、不可复制的部分

输出：

- 可复制
- 半可复制
- 高门槛 / 不可复制

## 12. 建议的正文结构

Skill 最终生成正式报告时，建议默认使用以下结构：

1. 案例摘要
2. 对象基本档案
3. 成长路径与职业迁移
4. IP 定位与内容策略
5. 渠道矩阵与传播方式
6. 商业模式与收入来源拆解
7. 关键决策节点复盘
8. 价值观 / 思维模型 / 方法论
9. 可复制性分析
10. 对目标用户的启发
11. 待验证信息与研究局限
12. 附录：时间线 / 证据库 / 链接清单 / 抽样表

## 13. 建议的技能资源拆分

按照 `$skill-creator` 的原则，这个 Skill 不应该把所有规则都塞进 `SKILL.md`。更合理的方式是将“触发与流程”留在主文件，把重型标准拆进 `references/`。

### 13.1 `SKILL.md` 中保留

- 技能用途与触发语境
- 默认输入与默认假设
- 执行流程摘要
- 输出契约
- 调研边界与强制规则
- 何时读取参考文件

### 13.2 `references/` 建议包含

- `research-standard.md`
  - 存放完整调研标准、证据等级、交叉验证规则、写作结构
- `report-template.md`
  - 存放标准报告结构模板
- `evidence-schema.md`
  - 存放证据库、时间线、抽样表、商业模式表的字段规范
- `search-playbook.md`
  - 存放检索关键词组合、平台搜索思路、抽样方法

### 13.3 `assets/` 建议包含

- `case-brief-template.md`
- `research-report-template.md`
- `evidence-table.csv`
- `timeline-table.csv`
- `content-sample-table.csv`
- `business-model-table.csv`

### 13.4 `scripts/` 建议

V1 可以先不强依赖脚本，因为核心能力是调研与分析，不是机械处理。

如果后续频繁重复搭建案例文件结构，可再考虑加入：

- `init_case_files.py`
  - 为一个新案例生成标准化研究文件模板
- `normalize_sources.py`
  - 清洗来源表字段或统一输出格式

## 14. 建议的技能实现重点

这个 Skill 真正有价值的，不是“会搜信息”，而是以下四件事做得稳定：

- 能先搭骨架，再填细节，而不是搜到哪里写到哪里
- 能持续区分事实、推断、未知
- 能把内容策略和商业模式连起来看
- 能提炼对超级个体学习者有用的方法结论

## 15. 质量标准

### 15.1 合格标准

- 能覆盖背景、内容、渠道、商业、关键节点、可复制性六大维度
- 有清晰时间线
- 有来源意识和证据标注
- 不把推测写成事实
- 商业模式分析不是口号式描述
- 输出结果对“研究者”有用，而不只是对“读者”友好

### 15.2 优秀标准

- 能识别关键转折点及其真实意义
- 能解释内容策略与商业闭环之间的关系
- 能区分表层人设与底层经营逻辑
- 能把启发落成具体动作，而非抽象原则
- 能在证据不足时主动收敛而不是硬写

## 16. V1 实现建议

建议按以下顺序推进：

1. 先基于本设计文档创建 Skill 文件夹
2. 将你现有的完整调研标准整理为 `references/research-standard.md`
3. 写一版精简但高触发质量的 `SKILL.md`
4. 补齐 2-4 个输出模板到 `assets/`
5. 用一个真实案例做首次 forward test
6. 根据首轮使用体验再决定是否增加脚本

## 17. 最终预期效果

这个 Skill 最终应该达到的效果是：

当用户只给出一个人物名或案例名时，Codex 也能沿着统一标准完成公开信息调研，输出一份中文、结构化、可追溯的案例研究结果；结果不仅能帮助用户“了解这个人”，更能帮助用户理解其内容策略、IP 路径、商业闭环，以及哪些经验真正值得超级个体学习。

换句话说，这个 Skill 的目标不是“写得像一篇文章”，而是“做得像一份研究产品”。
