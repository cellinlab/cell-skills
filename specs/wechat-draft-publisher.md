# 公众号草稿发布 Skill 设计文档

## 1. 目标概述

这个 Skill 用于把“已经写好的内容”稳定发布到微信公众号草稿箱，并在发布失败时给出可执行的故障定位与降级路径。

它解决的不是“帮用户写公众号文章”，而是帮助用户完成下面这条更窄、但更容易出真实故障的链路：

- 检查当前仓库是否已经有可复用的公众号发布实现
- 检查发布配置是否完整，包括 `appid`、`secret` 和 API IP 白名单
- 检查标题、摘要、正文图片、封面等是否满足微信发布约束
- 按正确顺序完成 token 获取、图片上传、封面上传、草稿创建
- 在发布失败时降级到本地 preview，而不是伪造“发布成功”

这个 Skill 的价值不在于“多会写内容”，而在于把最容易卡人的发布环节做成一个更稳的、可迁移的工作流。

## 2. 设计来源

这个 Skill 主要吸收了两个现成实现中“发布到公众号”的部分：

- `oaker-io/wewrite`
- `YouMind-OpenLab/skills/youmind-wechat-article`

保留的部分：

- 先检查配置与环境，再进入发布
- Markdown / HTML 转微信可接受内容
- 正文图上传、封面上传、草稿创建三段式发布流程
- 发布失败时提供本地 preview 降级
- 保留微信接口返回的 `errcode` / `errmsg`，不做模糊报错

改造的部分：

- 明确砍掉选题、写作、SEO、AI 配图、数据复盘等上下游能力
- 从“公众号全流程 Skill”改成“发布-only Skill”
- 更强调“优先复用目标仓库已有实现”，而不是默认重写
- 用 Cell 仓库更克制的方式定义边界，避免这个 Skill 重新膨胀成一个大而全工具

## 3. 技能定位

它是一个“公众号草稿发布 / 发布排查 / 发布降级” Skill。

它不是：

- 写作 Skill
- 选题 Skill
- SEO Skill
- 配图生成 Skill
- 公众号全流程运营 Skill

更准确地说，它处理的是“内容已基本完成，最后一公里如何稳定进草稿箱”。

## 4. 与现有 Skill 的关系

- `content-strategy-diagnosis`
  - 负责在开写前判断内容方向、形式和平台是否成立
- `deep-writer`
  - 负责深度内容的 Brief、结构和正文写作
- `xhs-image-director`
  - 负责平台内容里的视觉方向设计，不负责公众号 API 发布
- `wechat-draft-publisher`
  - 负责在内容已经存在的前提下，做发布前检查、草稿推送和失败排查

理想顺序通常是：

1. 先用策略类或写作类 Skill 产出内容
2. 再用 `wechat-draft-publisher` 完成草稿发布

如果用户实际上缺的是正文，而不是发布能力，应该切回写作链路，而不是在这个 Skill 里偷偷扩 scope。

## 5. 建议技能名

建议英文目录名：

`wechat-draft-publisher`

建议展示名：

`公众号草稿发布`

## 6. 适用场景

这个 Skill 主要适用于：

- “把这篇 Markdown 发到公众号草稿箱”
- “仓库里已经有发布脚本，但现在发不出去，帮我排查”
- “给这个项目补一个只负责公众号草稿发布的最小流程”
- “不要帮我写文章，只帮我把现成内容排版并发到草稿箱”

## 7. 非目标

V1 默认不解决以下问题：

- 从零写公众号文章
- 自动做选题、结构、写作和改稿
- 生成封面创意、插图创意或 AI 配图 prompt
- 分析阅读量、转发率、完读率等数据表现
- 替用户完成公众号后台的最终人工发布动作

## 8. 核心用户问题

这个 Skill 需要稳定回答以下问题：

1. 当前仓库有没有现成的公众号发布实现可以直接复用？
2. 这次不能发布，到底是配置问题、白名单问题、图片问题，还是 API 调用顺序问题？
3. 正文图片、封面图片和草稿创建分别该怎么处理，边界在哪里？
4. 什么情况下应该真实发布，什么情况下应该降级到本地 preview？
5. 在不改写正文的前提下，如何做发布所需的最小安全修正？

## 9. 典型触发语句

- 把这篇文章发到公众号草稿箱
- 帮我把 Markdown 发布到微信公众号
- 这个公众号发布脚本报错了，帮我看下
- 只做公众号发布的部分
- 给这个项目补一个 publish-only 的公众号 skill

## 10. 默认输入约定

### 10.1 必填输入

- 至少提供以下之一：
  - 一篇现成文章
  - 一个 Markdown / HTML 文件
  - 一个已有公众号发布实现但出现故障的项目

### 10.2 选填输入

- 标题覆盖值
- 摘要覆盖值
- 封面路径
- 作者名
- 主题 / 样式参数
- 当前报错信息

### 10.3 默认假设

如果用户没特别说明，默认：

- 输出语言为中文
- 用户要的是“发到草稿箱”，不是直接群发
- 不主动改写正文内容，只做发布必需的最小修正
- 优先复用仓库已有 CLI / script / SDK
- 如果缺真实发布条件，就明确降级到 preview

## 11. 输出契约

### 11.1 成功发布

至少包含：

- 最终标题
- 摘要
- 是否上传了封面
- 返回的 `media_id`
- 提醒用户去公众号后台草稿箱检查并手动发布

### 11.2 发布失败

至少包含：

- 失败发生在哪一步
- 关键报错信息
- 是否已生成本地 preview
- 还缺什么配置或人工动作

### 11.3 仓库改造型任务

如果用户要补发布能力，而不是只跑一次发布，至少包含：

- 当前项目已存在能力的复用判断
- 新增或修改的发布链路说明
- 预检、上传、草稿创建和 fallback 的边界
- 验证结果或未验证的风险点

## 12. 方法论原则

### 12.1 先看有没有现成实现，再决定改什么

公众号发布很容易“重复造轮子”。如果仓库里已有稳定实现，优先复用。

### 12.2 发布前检查优先于 API 重试

很多失败不是代码逻辑错误，而是配置不全、IP 白名单缺失、图片路径不对。不要一上来就重试。

### 12.3 正文图、封面图、草稿创建必须分层处理

正文图是正文资源，封面图是草稿字段依赖，草稿创建是最终落地动作。这三层不要混。

### 12.4 成功条件必须收敛

不能因为“接口没抛异常”就算成功。发布成功要以拿到 `media_id` 为准。

### 12.5 缺真实发布条件时要诚实降级

如果没有凭据、白名单没配、图片条件不成立，就生成 preview，并清楚告诉用户为什么还不能真实发布。

## 13. 资源规划

建议资源结构：

- `skills/wechat-draft-publisher/SKILL.md`
  - 核心工作流、输入输出契约、硬约束、资源导航
- `skills/wechat-draft-publisher/references/publish-runbook.md`
  - 发布链路搜索入口、上游共识、预检清单、API 顺序、常见故障与降级策略
- `skills/wechat-draft-publisher/agents/openai.yaml`
  - 展示名、短描述和默认调用提示

## 14. V1 关键约束

1. 不把这个 Skill 做成公众号全流程工具
2. 不在没有现成内容时偷偷接手写作任务
3. 必须优先检查并复用目标仓库已有发布实现
4. 必须保留 preview fallback
5. 必须把真实错误信息和发布边界说清楚
