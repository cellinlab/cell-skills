# X 内容策略 Skill 设计文档

## 1. 目标概述

这个 Skill 用于把 X/Twitter 上的内容工作，从“临时写一条推文”，升级为一套可持续运行的内容策略系统。

它要解决的不是“帮我随手写几条 tweet”，而是帮助用户稳定回答这些问题：

- 这个题到底适不适合发在 X
- 应该发短帖、Thread，还是先不写
- 当前问题在于选题、表达、节奏、增长动作，还是账号定位
- 如果要做账号诊断，应该采哪些数据、怎么看、如何沉淀成后续策略

## 2. 设计来源

这个 Skill 明显吸收了 `alchaincyf/x-mentor-skill` 的核心结构，但会改造成更适合 Cell 仓库的版本。

保留的部分：

- 渐进式披露，而不是把全部方法堆进一个文件
- 五场景路由：选题 / 写作 / 审阅 / 增长 / 账号诊断
- 账号诊断的数据采集、复盘、用户数据持久化思路
- 把“为什么这样做”的策略层逻辑显式写出来

改造的部分：

- 从“X 导师”强人设，改成 Cell 风格的诊断型、顾问型技能
- 不把具体平台权重写成僵硬真理，而是保留为“历史有效的高权重信号”，提醒按需验证
- 更强调和仓库内已有 skill 的分工，例如 `content-strategy-diagnosis`、`hook-optimizer`、`celf-style-writer`
- 更关注中文创作者、独立开发者、出海表达者的双语与跨语境场景

## 3. 技能定位

它是一个“X 内容策略 / 写作路由 / 增长诊断 / 账号复盘” Skill。

它不是：

- 通用 tweet 生成器
- 只讲算法玄学的增长模板
- 替代完整品牌策略或商业咨询的万能顾问

## 4. 与现有 Skill 的关系

- `content-strategy-diagnosis`
  - 负责先判断题目值不值得做、适合什么载体
- `x-content-mentor`
  - 负责进入 X 语境后，做场景路由、写法、增长动作和账号诊断
- `hook-optimizer`
  - 负责已经决定写之后，继续压前两行和开头抓力
- `celf-style-writer`
  - 负责把内容进一步调成 Cell 自身的表达风格
- `benchmark-filter`
  - 负责找值得研究的对标账号或创作者

## 5. 建议技能名

建议英文目录名：

`x-content-mentor`

建议展示名：

`X 内容策略`

## 6. 适用场景

这个 Skill 主要适用于：

- “我这个项目怎么发到 X 上更成立”
- “帮我想 X 上该发什么”
- “这条 tweet / thread 帮我审一下”
- “为什么我在 X 上涨不动”
- “分析一下我的 X 账号”

## 7. 非目标

V1 默认不解决以下问题：

- 只凭一句模糊需求，批量生成几十条空心推文
- 对具体平台算法做高置信度、实时性的断言
- 代替浏览器自动化本身去保证抓到所有 X 数据
- 脱离用户定位和长期目标，只追求短期爆帖

## 8. 核心用户问题

这个 Skill 需要稳定回答以下问题：

1. 当前到底该先做选题、写作、审阅、增长，还是诊断
2. 怎样把一个项目、经验或观点转成 X 原生内容
3. 为什么这条内容发不起来，是题不对、写法不对，还是账号结构不对
4. 账号诊断的数据应该怎么采、怎么读、怎么留存
5. 哪些建议属于高置信度共识，哪些只是阶段性经验

## 9. 典型触发语句

- 帮我写一条推文
- 这个项目怎么发 X
- 我在 X 上应该发什么
- 帮我看下这条 tweet
- 我账号最近涨不动了
- 分析一下我的 X 账号

## 10. 默认输入约定

### 10.1 必填输入

- 至少提供以下之一：
  - 一个选题或项目
  - 一条现有帖子
  - 一个账号名
  - 一个增长问题

### 10.2 选填输入

- 目标受众
- 语言：中文 / 英文 / 双语
- 内容目标：增长 / 建认知 / 导流 / 招聘 / 产品验证
- 当前账号阶段：新号 / 冷启动 / 已有一定受众

### 10.3 默认假设

如果用户没说明，默认：

- 输出语言用中文
- 如果最终帖子面向英文 X，帖子正文用英文，解释与策略用中文
- 先判断场景，再决定写不写
- 优先给一条可执行下一步，而不是泛泛原则

## 11. 输出契约

### 11.1 标准版

至少包含：

- 当前场景判断
- 目标受众与内容目的
- 主要问题或主要机会
- 推荐动作
- 如适用，给出可直接测试的帖文版本

### 11.2 账号诊断版

至少包含：

- 样本说明
- 核心指标概览
- 内容类型 / 时间段 /互动结构观察
- 主要结构性问题
- 下阶段动作清单
- 用户数据留存建议

## 12. 方法论原则

### 12.1 先判断场景，再动笔

不是每个问题都应该直接写 tweet。

### 12.2 X 上的内容是“题 + 包装 + 互动设计”的组合

只改文案，不改话题和互动结构，通常提升有限。

### 12.3 历史高权重信号值得参考，但不能当永恒真理

回复、书签、主页点击、停留时长通常比点赞更有价值，但具体权重会变化。

### 12.4 诊断要能沉淀

如果做过账号诊断，后面所有建议都应该利用这次沉淀，而不是每次从零开始。

### 12.5 输出要能和其他 Skill 接力

当真正的问题不是 X 写法本身时，要明确切给更适合的 Skill。

## 14. V1 关键约束

1. 不能把所有问题都降维成“写一条更好的 tweet”
2. 不能把上游 repo 的算法数字当作永久真理照抄
3. 必须保留账号诊断与用户数据持久化这一高价值能力
4. 必须明确与仓库内现有 skills 的协作边界
5. 必须让输出更像 Cell 的诊断式创作工作流，而不是平台培训课讲义
